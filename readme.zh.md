# open-llm-wiki

**自己会生长的个人 AI 知识库。**

灵感来自 [Andrej Karpathy 的 LLM Wiki 概念](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)。经过 23 篇论文实战验证。3 个 OpenClaw Skill。入库 → 质检 → 查询 → 演化。

---

## 问题

你读论文、记笔记，然后……一周就忘光了。

传统笔记是**只写不读**——你捕获了知识，但从不检索或连接。更糟的是，笔记不会生长。每读一篇论文，都从零开始。

## 洞察

Karpathy 的 LLM Wiki 思想：**知识库应该像复利一样增长**。

- 每入库一篇论文，都会丰富已有的概念（不只是多一个文件）
- 你问的每个问题都能让 wiki 生长（不只是消耗它）
- 矛盾会被自动检测（而不是被静默覆盖）
- 质量由独立的 AI 审查员保证（而不是自评）

**结果**：一个每次交互都变得更聪明的知识库，而不只是变得更大。

## 工作原理

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  论文    │────▶│  Ingest  │────▶│  独立QA  │────▶│  晋升    │
│  (PDF)   │     │  流水线  │     │ (子代理) │     │  stable  │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
                                                            │
                       ┌────────────────────────────────────┘
                       ▼
              ┌──────────────┐     ┌──────────────┐
              │   概念页     │────▶│  矛盾检测    │
              │   更新       │     │  (子代理)    │
              └──────────────┘     └──────────────┘
                       ▲
                       │
              ┌──────────────┐
              │   查询       │
              │   写回       │  ← 你的问题也在让 wiki 生长
              └──────────────┘
```

### 三条流水线

| 流水线 | 触发时机 | 做什么 |
|--------|---------|--------|
| **Ingest** | 你添加一篇论文 | 解析 → 起草 → 自检 → 独立QA → 修正 → 晋升 → 更新概念 → 矛盾检测 |
| **查询写回** | 你问关于 wiki 内容的问题 | 搜索 wiki → 评估覆盖 → 回答 → 有价值的合成写回 wiki |
| **Lint** | 每日（定时） | 格式合规、QA 覆盖、交叉引用完整性、日志健康 |

### 核心创新：独立 QA

**大模型不能自评。** 这是 23 篇论文教会我们的第一课：

- 自检能发现错别字，但发现不了错误数字、张冠李戴的数据、微妙的矛盾
- 每篇论文都经过**独立子代理 QA**（独立上下文、独立会话）
- QA 评分 ≥ 7.0 才能从草稿晋升为正式条目
- 矛盾检测同样使用独立子代理

## 快速开始

### 前提条件

- 已安装 [OpenClaw](https://github.com/openclaw/openclaw)
- 已配置 AI 模型（子代理推荐 GLM-5.1）

### 安装

```bash
# 克隆仓库
git clone https://github.com/yourusername/open-llm-wiki.git

# 复制 skills 到 OpenClaw skills 目录
cp -r open-llm-wiki/skills/* ~/.openclaw-autoclaw/skills/

# 创建 wiki 目录结构
mkdir -p my-wiki/{raw,sources,concepts,drafts,qa-reports,templates,_state,log-archive}

# 复制模板和 schema
cp open-llm-wiki/SCHEMA.md my-wiki/
cp open-llm-wiki/templates/* my-wiki/templates/
```

详见 [QUICKSTART.md](QUICKSTART.md)。

## 设计原则

### 1. 论文是原料，概念才是 wiki

论文只是原始材料。概念——持续演化的、交叉引用的理解——才是真正的知识库。一篇论文的入库应该更新 3-5 个概念页。

### 2. 大模型不能自评

自检能发现错别字。独立 QA 能发现错误数字、基准张冠李戴、微妙矛盾。QA 和矛盾检测都用独立子代理会话。

### 3. 查询让 wiki 生长

好问题产生好合成。一个对比表、一条时间线、一个你发现的关联——这些很有价值，不该消失在聊天记录里。

### 4. 矛盾标记，永不静默覆盖

当新证据与旧结论冲突时，两者都保留，用 `⚠️ [CONTRADICTION]` 标记。真理从辩论中产生，不从覆盖中产生。

### 5. 硬数字是脊梁

"模型取得了有竞争力的结果"是废话。"MiniF2F-test: 88.9% pass ratio, PutnamBench: 47/658"才是知识。我们遇到的每一次 QA 失败，都是因为缺少硬数字。

## 实战验证

本框架通过入库 **23 篇 DeepSeek 研究论文**（2024.01 – 2026.01）构建和验证，覆盖：

- 架构演化：DeepSeek LLM → V2 → V3 → V4
- 推理突破：R1、GRPO、蒸馏
- 多模态：VL、VL2、Janus 系列
- 专业化：Math、OCR、Prover、mHC

关键指标：
- 31% 首次 QA 通过率 → 教会我们先写硬数字
- claude-sonnet-4 子代理可靠性 0/3 → glm-5.1 达到 4/4
- 抓住 1 次关键数据张冠李戴（V3.2 Figure vs Table）
- 全部 23 篇论文与原文交叉验证
